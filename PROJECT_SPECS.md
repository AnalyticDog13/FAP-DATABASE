## Component build order

Each step ends with the Definition of Done (see CLAUDE.md) and a Risks & Reality Check. Do not move to step N+1 until step N has passed all DoD gates and the user has acknowledged the report.

| Step | What |
|---|---|
| 0 | Scaffold: directory structure, pyproject.toml, .env.example, docker-compose.yml, Dockerfile, base FastAPI app with /health, Alembic init, empty __init__.py files |
| 1 | Database + first migration: ORM models, first Alembic migration, db/session.py with async engine |
| 2 | Philadelphia seed list: seed_philly.py produces philly_hospitals.csv (manually curated for v1) |
| 3 | IRS Schedule H ingestion: irs_990.py — find filings, download XML, parse Schedule H Part V Sec B, upsert |
| 4 | FAP document scraper: fap_scraper.py — HEAD, GET, SHA-256, change detection, change_log writes |
| 5 | PDF text extraction: pdf_to_text.py — pdfplumber primary, pypdf fallback, scanned-PDF detection |
| 6 | LLM extraction: llm_extract.py + prompts.py — Anthropic SDK with tool-use forcing, temperature=0 |
| 7 | Reconciliation: reconcile.py — Schedule H wins on conflict, disagreements logged |
| 8 | API: all endpoints, Pydantic schemas, auth, OpenAPI docs — hit every endpoint live before declaring done |
| 9 | Local scheduler (dev only): APScheduler in scheduler/local_runner.py |
| 10 | Production scheduling: GitHub Actions nightly_fap_check.yml + Claude Code Routine |
| 11 | Tests + CI: real tests (not stubs), ci.yml with Postgres service container |
| 12 | README: architecture diagram, quickstart, API reference, limitations/disclaimers |

### Production scheduling split (Step 10)

**GitHub Actions** (nightly_fap_check.yml, runs Monday 09:00 ET): deterministic work — download, hash, detect changes, write to `pending_extractions` queue table.

**Claude Code Routine** (update_faps.md, runs Monday 10:00 ET): AI-judgment work — pull pending_extractions, run PDF extraction via Anthropic API, write fap_extractions, write change_log, post summary.

Rationale: GHA is free and reliable for deterministic hashing/diffing. Routines handle LLM reasoning where observability matters. Do not put deterministic work in Routines (counts against daily cap) and do not put LLM work in GHA without it being the right tradeoff.

## What NOT to do

- Don't write `requirements.txt` — use `pyproject.toml` with `[project.dependencies]`
- Don't write `schema.sql` — use Alembic migrations
- Don't use psycopg2-binary directly — use SQLAlchemy 2.0 async with `asyncpg` driver
- Don't use MD5 — use SHA-256
- Don't hardcode the Philadelphia hospital list inside Python — keep it in `data/seed/philly_hospitals.csv`
- Don't make `npi` the primary key — use UUID surrogate
- Don't use the model name `claude-opus-4-20250514` — that's outdated; use `claude-opus-4-7`
- Don't build all five components in parallel — go end-to-end on one hospital first
- Don't omit the `change_log` table — it's the most valuable feature for B2B customers
- Don't declare a step "done" without running the Definition of Done checklist
- Don't agree with the user when the user is wrong; cite the rule in CLAUDE.md and explain
